# Conventions

Coding patterns, naming rules, and style agreements for this project.
Follow these without question. Do not deviate unless explicitly told.

## Naming
- Prefix semantic theme variables with `--CTBS-` and use PascalCase names
- Generate an `Rgb` companion for each base CTBS color variable for `rgba()` usage
- Use `DarkTheme` prefixes for dark-mode role families such as `--CTBS-DarkThemePrimary`
- Use `{Role}`, `{Role}Rgb`, `{Role}BtnColor`, `{Role}BtnBg`, `{Role}TextEmphasis`, and `{Role}BgSubtle` naming patterns to mirror Bootstrap semantic surfaces
- Reserve `BodyBg`, `BodyColor`, and `EmphasisColor` for page-level foreground and background semantics

## File Layout
- Treat `extract_bootstrap_colors.py`, `ColorSim.py`, `index.html`, source images, and `bs/bootstrap-5.3.8.css` as committed inputs
- Treat `bs/bootstrap-overrides.css`, `bs/ctbs-variables.css`, and `bs/*-theme.css` as generated outputs regenerated by `generate_all.sh`
- Load browser CSS in this order: `bs/bootstrap-5.3.8.css`, `bs/ui-config.css`, `bs/<theme>-theme.css`, `bs/bootstrap-overrides.css`
- Keep the generation pipeline as `extract_bootstrap_colors.py` first, then `ColorSim.py` once per theme, with `generate_all.sh` orchestrating the full run
- Keep `index.html` as the comprehensive Bootstrap preview catalogue and manual verification surface for theme behavior

## API Patterns
- Register synthetic dark role variables for all 8 Bootstrap roles so dark utilities resolve through `--bs-{role}-rgb`
- Prefer data-driven selector and property tables for repeated override injection logic instead of long manual branch chains
- Structure `bootstrap-overrides.css` in five layers: base Bootstrap variable remap, component overrides, accessibility safety overrides, `.text-bg-*` contrast fixes, and dark outline button overrides
- Generate dedicated dark-mode companion rules for every glass selector and dedicated dark injections for alerts and contextual tables
- Pair text variables against their rendered background through `background_pair_for()` before applying contrast correction

## UI Patterns
- Apply glass treatment only to container and chrome components such as cards, modal content, navbars, dropdown menus, list-group items, toasts, offcanvas panels, `.content-wrapper`, `.card-header`, and `.card-footer`
- Keep alerts and tables opaque even when other container surfaces use glassmorphism
- Handle accordion translucency through Bootstrap variable overrides such as `--bs-accordion-bg` instead of putting accordion selectors into the glass selector list
- Render the page background image through `body::before` with fixed full-viewport coverage on desktop browsers
- Store the active theme in `localStorage` under `ct-theme` and the active mode under `ct-mode`
- For bundled themes, switch the loaded theme CSS and the mapped background image together from the preview page controls

## Database
- No database is used in this project

## Testing
- Activate `venv` before running repository Python scripts
- Run `make generate` or `./generate_all.sh` before tests that read generated CSS assets
- Evaluate WCAG checks with glass opacity forced to `1`, background imagery removed, and transitions disabled
- Use `test_contrast.py` for generated-variable regression checks, `test_browser_wcag.py` for rendered Playwright validation, and `browser_wcag_tool.py` for standalone audits
- Expect browser tests to serve the project over local HTTP instead of opening `index.html` directly from disk

## Operations
- Use `make generate` for the full extractor-plus-theme regeneration flow and `make test`, `make test-contrast`, `make test-browser`, and `make test-audit` for the standard verification entry points
- Install Playwright Chromium with `python -m playwright install chromium` when browser-based WCAG checks are needed
- Keep the GitHub Pages workflow responsible for generating theme CSS before publishing the preview site

## Code Map
- `extract_bootstrap_colors.py` owns Bootstrap parsing, CTBS variable registration, override generation, glass injection, dark alert and table injection, and accessibility tail fixes
- `ColorSim.py` owns image palette extraction, role mapping, contrast correction, background pairing, and final theme CSS emission
- `test_contrast.py` checks generated variable relationships without a browser, while `test_browser_wcag.py` and `browser_wcag_tool.py` validate the rendered preview surface
- `index.html` is the manual verification surface for theme switching, mode switching, glass controls, and Bootstrap component coverage
